# FamClaw / Butler — Roadmap (2026-05-12)

Captured from a strategy session on 2026-05-12 after diagnosing a v0.5.9 production failure on the Mac Mini. Covers: current bug analysis, sibling-claw comparison (OpenClaw, PicoClaw), adoption decisions, prioritized phased roadmap through Phase 5, and explicit non-goals.

This is a strategy doc. Each phase item gets its own spec (in `specs/`) and implementation plan (in `plans/`) before code is written.

## 1. What triggered this

On `2026-05-12 09:01-09:05`, a parent user asked Butler for travel research. Butler 400'd four times in a row on Discord with:

```
LLM error in tool loop iteration 1: LLM returned 400:
{"error":{"message":"request (23364 tokens) exceeds the available context size (16384 tokens)..."}}
```

Root cause: `web_fetch` returns up to ~256 KB of extracted page text. That text is pasted verbatim into the tool-loop's secondary LLM call (`internal/agentcore/stage_tool_loop.go:225`). Nemotron's llama-server is configured at `--ctx-size 16384`, so a single fat page overflows the window on its first round-trip.

Why this just started happening: v0.5.9 made `spawn_agent` work for the first time. Before, big payloads were absorbed by a subagent and only a short string reached the parent. Now Nemotron also calls `web_fetch` directly from the top-level loop, putting the full payload in parent context.

Secondary observations from the same logs:
- `url_allowlist` blocks many family-relevant hosts (`visitstpete.com`, `stpete.org`, `neworleans.com`, `tripadvisor.com`, `timeanddate.com`, `forecast.weather.gov`, `nola.com`), causing the agent to flail through 3-4 retries.
- `spawn_agent` subagent results sometimes return 0 chars (07:55 and 08:39 in logs). Separate bug.
- Model sometimes calls tools by bare name (`web_fetch`) instead of `builtin__web_fetch`. Dispatch may be failing silently.

## 2. The family-assistant filter

Butler is **not** a generic AI agent. It's specifically a family bot serving two parent users and one teen user (age_13_17). Every roadmap decision is filtered through:

- Does this serve real family moments — research, reminders, homework, household questions?
- Does it work across the day, across people, across devices (TG / WA / Discord / web)?
- Does it stay safe for the teen without being annoying for the parents?
- Does it preserve famclaw's defining invariants (OPA on every message, identity-resolve-before-LLM, age-aware roles, single Go binary, local-first)?

Decisions that look architecturally right but fail the filter (e.g., SearXNG sidecar, full coding-agent capability) get cut.

---

## 3. FamClaw's defining moat — protect, don't compromise

The following stack is what makes Butler safe and family-appropriate where OpenClaw and PicoClaw are not. Nothing in this roadmap compromises it:

- **OPA policy gate on every message** before the LLM sees it
- **Identity-resolved-before-LLM** — no anonymous traffic
- **Age-aware roles** + parental approval workflow for restricted topics
- **Local-first, single Go binary, CGO_ENABLED=0** — runs on Pi 3/4/5, Mac Mini, Termux, Windows
- **AgentSkills (SKILL.md)** + HoneyBadger security scanning

These are unique to famclaw. Neither sibling has equivalents.

### Clarification: "pi" is not Raspberry Pi

The "pi" in OpenClaw's `docs/pi.md` refers to **pi-mono / pi-coding-agent** (`@earendil-works/pi-coding-agent`) — a TypeScript embeddable AI agent SDK by Mario Zechner (badlogic). OpenClaw embeds it. There's no Raspberry-Pi-flavored CLI in either project.

Famclaw cannot import pi-coding-agent: wrong language, would require a Node runtime on every Pi/Mac/Termux box, kills the single-binary CGO=0 invariant. Adopting means stealing patterns, not code.

## 5. Adoption decisions

### Steal from PicoClaw (Go, same language, MIT)

| Item | What to take | Why |
|---|---|---|
| `pkg/cron` | Code patterns or direct port | Highest-value missing family feature: reminders. |
| `pkg/tokenizer` | Patterns | Better than `chars/4` for budget math, if we ever need accuracy. |
| `pkg/updater` | Code patterns | Fixes the manual Mac Mini deploy mess (memory: `reference_mac_mini_deploy`). |
| `pkg/agent` SubTurn/Hooks/Steering/EventBus | Patterns only | Worth reading before next agentcore refactor. |
| `cmd/picoclaw/internal/onboard` | UX shape | Optional TUI onboarding to complement web UI. |

**Do not** fork PicoClaw. Their threat model (single-user embedded device) doesn't carry famclaw's policy/identity invariants.

### Steal from OpenClaw (TypeScript, different stack — read the docs)

| Item | What to take | Why |
|---|---|---|
| Compaction safeguard mode | Pattern (Go rewrite) | Adaptive token budgeting + tool-failure summaries when context fills. |
| Cache-TTL context pruning | Pattern (Go rewrite) | Prunes old turns by cache-touch age; `isToolPrunable` predicate drops tool results before conversation. |
| Overflow error classifier | Pattern | OpenClaw enumerates ~7 overflow signatures across providers including `ollama error: context length exceeded`. Port to detect+retry. |
| Browser tool architecture | Pattern (MCP sidecar) | Don't embed Chromium in famclaw. Use chromedp Go MCP server. |
| `docs/pi.md` itself | Read it as a free design doc | Best reference for compaction + pruning + tool policy in a real embedded agent. |

**Do not** fork OpenClaw. Different language, different threat model, different product (cloud-deployable mobile + macOS).

### Skip from both

- Multi-profile cloud auth + failover (no use until famclaw has a cloud escape-hatch; local-first is the identity)
- Sandbox / container exec subsystem (overkill — Tier-B workspace tools are the right granularity, see Phase 4)
- TUI mode (families don't use terminals)
- More gateways (Matrix, IRC, Slack, WeChat) — Butler covers what families use
- LSP integration — Butler is not a coding agent

## 6. Coding-agent capability — clarified position

After pushback on overgeneralized risk framing: Butler's OPA layer is precisely what makes coding capability safer here than on Crush/Cursor/Claude Code. Identity is known, role is known, topics are gated.

Three distinct capability tiers, treated separately:

- **Tier A — In-conversation code help** ("explain this Python," "help me debug this React"). Already works because the LLM is the LLM. OPA gates it. Audit the policy for code-help topics per role. **In scope, small.**
- **Tier B — Workspace-scoped code tools.** Per-user workspace (`~/butler-workspace/<user>/`) with OPA-gated `read_workspace_file`, `write_workspace_file`, `run_in_workspace`. Parent-visible audit trail. **In scope, larger — Phase 4.**
- **Tier C — Full IDE coding-agent verbs** on the host filesystem. **Out of scope.** Not a safety call — a product call. If parents want Claude Code, they have Claude Code.

## 10. Risks

Threats to this plan that have to be managed in per-feature specs:

- **Spillover cache retention (privacy).** Phase 2 stores 24h of fetched page text + subagent output per user on disk. If a teen-role user researches sensitive topics (mental health, sexuality, school conflicts), that content sits on the host for a day. Mitigations: tighter default TTL for child roles (e.g., 1h), per-user size cap, cache encryption at rest if disk is shared, parent-visible audit policy that respects child privacy (parents see *categories*, not full text, until a hard-block fires).
- **Summarization erasing context the family relied on.** Phase 2 Tier 1 compresses old turns. If the summary drops "parent X is allergic to penicillin" mentioned three days ago and never re-pinned, a later "what should that parent take for a sinus infection" misses it. Mitigations: family_state table (Phase 3.3) is the load-bearing solution — durable facts live there, not in conversation history. Summarizer prompt explicitly preserves named-entity claims + dates.
- **OPA policy drift as tool surface grows.** Each new tool (web_fetch, spawn_agent, tool_result_more, cron, message_route, browser navigate/click/evaluate, workspace read/write/run, vision) is a new OPA gate. Combinatorial complexity. Risk: a policy passes individual unit tests but lets a multi-tool chain through. Mitigations: tool-chain integration tests, deny-by-default for new tools, periodic policy audit.
- **Tier-B execution sandboxing.** OPA gates the *call* — it does not sandbox the *execution* of `run_in_workspace`. A teen who can call `run_in_workspace` to run python in `~/butler-workspace/<their-user>/` can also run python that exfiltrates env vars or makes network calls. Mitigations TBD in Phase 4.2 spec: container? bubblewrap? plain user-namespace? cross-platform implications are significant (Mac vs Linux vs Termux). If sandboxing is heavy, Tier B slides to Phase 5.
- **Reactive recovery loop (Phase 2.6).** If Tier 1 summarization itself overflows context, the retry overflows again. Mitigations: hard cap on retries (exactly one), distinct prompt budget for the summarization call vs the user-facing call, fallback to Tier 0 truncation + return-with-warning to the user if recovery fails twice.
- **Single-binary update integrity.** Until OTA updater (Phase 4.3) lands, deploys are manual scp + nohup. No checksum verification on the wire by default — the binary you push *is* the binary that runs, with no MITM guard. Memory entry `reference_mac_mini_deploy` documents a shasum step but it's manual discipline, not enforced. Mitigations: prioritize Phase 4.3 OTA, in the meantime keep the manual checksum step in deploy scripts.
- **Local LLM single-point-of-failure.** If the Nemotron endpoint is down (model crash, GPU hang, llama-swap stuck), Butler is down. By design — local-first means no cloud fallback. Mitigations: better health-checks + clearer user-facing error when the endpoint is unreachable ("Butler's brain is rebooting, try again in a minute") rather than a raw 5xx; consider a tiny on-device fallback model for "I'm sorry, I'm down" responses; revisit if outages become common.
- **Sqlite WAL growth under heavy cache use.** spillover cache writes 256KB blobs (in disk-split design, smaller index rows). High research-day volume could grow the DB faster than the TTL sweep can collect. Mitigations: monitor DB size on the host, tune sweep interval, consider VACUUM on a schedule.
- **Empty tool/subagent results killing user trust.** Already happening (`completed (0 chars)` log lines). Per-spec UX commitment: Butler never returns empty content to the user — if a tool produced nothing, surface a brief user-readable message ("I tried to look that up but came back empty-handed — want me to try a different angle?") instead of silence.

Each per-feature spec gets a Risks section that inherits and refines these.

## 11. UX commitments

Roadmap is engineering-shaped. These are the family-facing UX invariants every per-feature spec must respect:

- **Never silent failure.** Butler responds to every message it processed, even when a tool failed or returned empty. No 0-char Discord replies. No raw 5xx error strings to the user.
- **Parent-approval surfaces in the same gateway.** If a teen asks something on Discord that needs a parent's approval, the approval request goes to that parent's preferred gateway (per identity table), the answer routes back to the teen's Discord. No "log into the web UI" friction.
- **Reminder delivery routes to the right person.** "Remind parent about the teen's dentist Thursday 4pm" reaches the named recipient on their preferred channel, not the chat where the reminder was set.
- **Streaming when there's no tool call.** Already works; preserve it. Long thinking pauses are bad family UX.
- **Cross-platform parity.** Any feature must work on Pi 3/4/5, Mac Mini, Termux, Windows. Features that work only on one platform get cut or deferred until they work on all four. The single-binary discipline enforces this.
- **Time-of-day-aware delivery for reminders/notifications.** Don't ping the teen at 11pm unless it's marked urgent. Schedule has knowledge of household quiet hours.
- **Audit-visible without being parental-snooping.** Parent oversight (Phase 5.2) surfaces *categories* and *policy decisions* by default, not full conversation transcripts. Full content is revealed only when a hard-block fires or when parent explicitly drills in with cause. Protects teen privacy while preserving safety.
- **Empty-state hospitality.** First-run, no skills installed, no LLM configured — every empty state has a clear next-step that a non-technical parent can act on, not a Go error message.

Per-feature specs cite these. Violations require explicit justification in the spec.
