## PR 1: `refactor/llm-openai-compat` — LLM Client Multi-Backend

**Why:** Unblocks cloud providers (Groq, OpenAI, OpenRouter) and llama.cpp server. The single most impactful change — every wizard preset actually works after this.

- `internal/llm/client_test.go` — rewrite

6. Add `ChatWithTools()` method — non-streaming with tool schemas, returns `Message` with `tool_calls`
7. Ollama also supports `/v1/chat/completions` since v0.1.24 — test both paths

**Test:**

- Table-driven: Groq URL, OpenAI URL, Ollama URL, llama.cpp URL → correct endpoint selection

## PR 6: `feat/context-compression` — Tiered Context Management

**Why:** Replace hard "last 20 messages" cutoff. Enable long conversations on RPi without OOM.

**Files:**
- `internal/compress/estimator.go` — new (TokenEstimator interface + SimpleEstimator)
- `internal/compress/truncate.go` — new (Tier 0: smart truncation)
- `internal/compress/summarize.go` — new (Tier 1: LLM summary compaction)
- `internal/compress/compress.go` — new (Compress function, orchestrates tiers)
- `internal/compress/compress_test.go` — new
- `internal/agentcore/stage_compress.go` — new (pipeline stage, runs before llm_call)
- `internal/store/db.go` — add `conversation_summaries` table

**Changes:**
1. `SimpleEstimator`: `len(text) / 4` for English
2. Tier 0 (70% of context): drop oldest turns, never drop system prompt / tool results / approvals
3. Tier 1 (when Tier 0 loses important context): call LLM to summarize, replace dropped messages
4. Tier 2 (emergency): summarize everything, keep system + summary + last 3 turns
5. `StageCompress` — inserts before `StageLLMCall` in FamilyPipeline
6. Context window: configurable `context_window` in LLMProfile, default 4096
7. `conversation_summaries` table for cross-session continuity

**Test:**
- Tier 0: 30 messages, 4K window → verify oldest dropped, system prompt kept
- Tier 1: 30 messages with tool results → verify summary generated, tool results preserved
- Tier 2: massive conversation → verify reset to summary + last 3
- Estimator: known strings → verify ~4 chars/token estimate

## PR 8: `feat/smart-tool-selection` — Token-Budget Tool Filtering

**Why:** Small models (4B on RPi) can't handle 30 tool schemas. Smart selection keeps tools within context budget.

**Files:**
- `internal/toolreg/selector.go` — new
- `internal/toolreg/selector_test.go` — new
- `internal/agentcore/stage_filter_tools.go` — update to use selector

**Changes:**
1. Strategy 1 (always): role filter — already in PR 3
2. Strategy 2 (always): skill-scoped tools — already in PR 3
3. Strategy 3 (large tool sets): two-pass classification
   - Send LLM a tool index (name + 1-line description, ~5 tokens each)
   - LLM responds with which tools it needs
   - Inject full schemas only for selected tools
4. Token budget ratio: 4K=500 tokens, 8K=1500, 32K+=8000, 128K+=unlimited
5. Conversation-only bypass: if classifier detects pure chat, skip all tool injection
6. Zero-tool fallback: if two-pass returns nothing, proceed without tools

**Test:**
- 40 tools, role filter → 10 tools → budget OK → no two-pass needed
- 40 tools, budget exceeded → two-pass mock LLM → 3 tools selected
- Pure conversation message → zero tools injected
- Token budget calculation for various context windows

**Docs:** None (internal optimization).

**Verify:** `go test ./internal/toolreg/... -v && go test ./internal/agentcore/... -v`

## PR 13: `feat/llama-server-sidecar` — llama.cpp Native Inference

**Why:** Remove Ollama dependency. FamClaw spawns `llama-server` directly, downloads GGUF models, supports TurboQuant builds for long context on constrained hardware.

**Files:**
- `internal/inference/sidecar.go` — new (spawns/manages llama-server process)
- `internal/inference/models.go` — new (model catalog + download)
- `internal/inference/sidecar_test.go` — new
- `cmd/famclaw/main.go` — integrate sidecar startup
- `scripts/install-rpi.sh` — download llama-server binary instead of Ollama

**Changes:**
1. `Sidecar` struct: manages `llama-server` subprocess lifecycle
   - `Start(ctx, modelPath, port, opts)` — spawn with GGUF model
   - `Stop()` — graceful shutdown
   - `Healthy()` — check `/health` endpoint
   - Auto-restart on crash with backoff
2. Model catalog: embedded JSON of recommended GGUF models per hardware tier
   - Tiny (1-2B, Q4_K_M): RPi 3, old Android — `qwen2.5-1.5b`
   - Small (4B, Q4_K_M): RPi 4 — `qwen3-4b`, `phi4-mini`
   - Medium (8B, Q4_K_M): RPi 5 8GB, Mac — `gemma4-e2b`, `llama3.1-8b`
3. Model download: fetch GGUF from HuggingFace, verify SHA256, store in `~/.famclaw/models/`
4. TurboQuant support: if `turboquant_plus` build of llama-server detected, enable KV cache compression flag (`--cache-type-k q4_0`)
5. Config: `inference.backend: "llama-server" | "ollama" | "external"`, `inference.model_path`, `inference.gpu_layers`
6. `main.go`: if backend is `llama-server`, start sidecar before web server, health-check before accepting connections
7. Wizard Step 4: add "Download and run locally" option that fetches model + starts sidecar

**Test:**
- Sidecar start/stop with mock binary
- Health check polling
- Model download with mock HTTP server → verify SHA256 check
- Config: `backend: external` → no sidecar started

**Docs:** Update README, CLAUDE.md, docs/HARDWARE.md with llama.cpp info. Update install-rpi.sh.
